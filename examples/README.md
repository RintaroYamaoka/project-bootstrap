# examples/

このディレクトリは、`project-bootstrap` の原則と TDD ワークフローが **実セッションでどう動いたか** を記録する場所。

## 目的

抽象的な原則 (SKILL.md) と subagent / command の定義 (`agents/`, `commands/`) だけでは、**実際の動き方** が伝わらない。実例があると:

- 新しい開発者がワークフローを掴みやすい
- AI が再現すべきパターンの「正解例」になる
- 期待通りに動かなかったセッションは、原則 / プリミティブの不足を示す材料になる

## 何を記録するか

各セッションログは、1 タスクに対する **`/plan` → Red → Green → Refactor → Close** の流れを 1 ファイルに収める (`TEMPLATE.md` 参照)。

含める情報:

- タスクの要旨
- `/plan` が出力した計画書
- Red フェーズ: 書いた failing テスト
- Green フェーズ: 通した最小実装
- Refactor フェーズ: 適用した改善 / 戻した変更
- Close: 確認したこと、残課題
- 振り返り: うまくいった点 / 困った点 / 原則・プリミティブへの気付き

## 命名規則

```
YYYY-MM-DD-<short-task-slug>.md
```

例: `2026-04-26-add-email-validation.md`

## 現在の収録セッション

なし。`/plan` skill と TDD subagent 群が実プロジェクトで稼働するに従って蓄積していく。

## 追加方法

1. `TEMPLATE.md` をコピー → 命名規則に従ってリネーム
2. セッション中の入出力をできるだけ忠実に貼り付ける (要約しすぎない)
3. 末尾の「振り返り」を埋める
4. PR にする (一度書いたら原則・プリミティブの改善材料として活用)

## 注意

- **実セッションのみ記録する**。フィクションのログは AI を誤誘導する。
- 機密情報 (社内コード・顧客データ) は含めない。汎用的なドメインに置き換えるか、そのセッションは記録しない。
- うまくいかなかったセッションも記録する。失敗例こそ原則改善の材料。
