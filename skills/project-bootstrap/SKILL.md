# AI 駆動開発の憲法とフロー

このスキルは、このプロジェクトにおけるコーディング作業の進め方を定義する。
プロジェクト立ち上げ時だけでなく、**機能追加・バグ修正・リファクタ・調査など、あらゆる作業**に適用する。

構成は三層:

- **Part 1 — 憲法**: プロジェクトを通じて変わらない原則。
- **Part 2 — 開発フロー**: TDD の Red-Green-Refactor を軸とする日常の作業手順。
- **Part 3 — AI 協働ルール**: Anthropic / Cursor の実運用に倣った AI エージェントとの協働方法。
- **Part 4 — テスト方針**: 振る舞い仕様としてのテストの書き方。

哲学:

- **コードを唯一の正解とする**。中間ドキュメント形式の仕様は持たない。仕様はテストで表現する。
- **小さく検証可能なステップ**を積み重ね、コードベースを常に理解可能な状態に保つ。
- **形式的なチェックリスト**としてではなく、自然に沿って動くこと。

## Part 1: 憲法 — 不変の原則

### Code is Truth — コードを唯一の正解とする

ドキュメント形式の仕様（要件定義書・詳細設計書・Confluence 仕様）は、実装が進むと必ずコードと乖離する。最終的に動いているコードが正解になる。

このプロジェクトは次の立場を取る:

- 振る舞い仕様は **テストコード** で表現する。テストはドリフトしたら即 fail する executable contract。
- 不変の原則は **この `SKILL.md`** に置く。これは滅多に変わらない憲法層。
- 個別の API / モジュール契約は **型定義 + テスト** で表現する。
- 別形式の仕様ドキュメントを正規ソースとして持たない。

正規ソースは三点だけ:

1. `SKILL.md` — 不変の原則
2. テストコード — 変動する振る舞い仕様
3. 実装コード — 動作する成果物

### 基本姿勢

- **書く前に理解する**。まずコードを読む。
- **信じる前に検証する**。API・ファイル・型が想定通り存在するかを確認する。
- **大きくやる前に小さくやる**。1 変更 = 1 責務。
- **完了を主張する前に実行する**。ビルド通過 ≠ 動作している。
- **症状より先に根本原因**。バグを小手先で回避しない。
- **フォールバックより先に Fail fast**。防御的コードで問題を隠さない。
- **新パターンより先に既存パターン**。明示的に変更する場合を除き、コードベースに合わせる。
- **実装より先にテスト**。AI は放っておくと実装を先に書く。明示的に逆をやる。

### コーディング原則 — 古典

#### KISS — Keep It Simple, Stupid

可能な最も単純な解を選ぶ。複雑さは正当化されたときだけ追加する。

- 巧妙な短い解より、明瞭な短い解。
- 抽象化は重複が **3 回以上** 現れてから（Rule of Three）。
- 「将来こうなるかも」を理由にしたオプション・分岐・パラメータを増やさない。

#### YAGNI — You Aren't Gonna Need It

仮想の将来要件のためにコードを書かない。

- 「いつか使うかも」のフィールド・引数・抽象を作らない。
- 「念のため」の互換レイヤー・フィーチャーフラグを置かない。
- 仮想ユースケースのための分岐 / オプションを追加しない。
- 必要になった時点で書く。それまでは存在しない方が安全。

#### DRY — 慎重に

重複は責務の重複であって行の重複ではない。

- 行が同じでも責務が違うなら、共通化しない。
- 違う理由で変更されるべきコードは、別々であるべき。
- 「とりあえず DRY」は「早すぎる抽象化」を生む。早すぎる抽象化はコピペより悪い。
- 3 回現れたら統合を検討する。2 回ならまだコピペでよい。

#### Fail Fast

前提が崩れた瞬間に大声で失敗させる。

- 不正な入力を検出したら即例外を上げる。
- 「動かし続けるため」だけのフォールバックを書かない。
- 想定外を握り潰す catch を書かない。
- 副作用系の失敗を catch するのは、主処理を安全に継続させる必要があるときだけ、かつ catch 範囲は最小に。

#### Root Cause

症状ではなく原因を直す。

- sort / filter / fallback / retry を追加する前に、発生源が間違っていないかを疑う。
- 共通コードを触らないための処理の複製は、構造が間違っているサイン。
- やむを得ず回避策を入れるなら、一時的である旨と理由をコメントで明記する。

#### Composition over Inheritance

継承ではなく合成を使う。

- 継承は「is-a」が真でなければ使わない。
- 振る舞いの再利用は委譲（合成）で行う。
- 継承階層が 2 段を超えたら、合成に置き換えられないか疑う。

#### Law of Demeter — 近接の法則

オブジェクトは直接の友達としか話さない。

- `a.b().c().d()` のチェーンは設計のサイン。
- 必要な情報は引数で受け取る。グローバル / 親オブジェクトを覗き見ない。

### SOLID 原則

#### S — Single Responsibility Principle

1 つのモジュール / 関数 / クラスは、変更される理由を 1 つだけ持つ。

- 「変換」「判断」「永続化」「表示」を 1 つのユニットに混ぜない。
- 「と」「、」を使って責務を説明している時点で、責務が分裂している。

#### O — Open/Closed Principle

拡張に対して開き、修正に対して閉じる。

- 既存コードを書き換えずに新しい振る舞いを追加できる構造を目指す。
- ただし YAGNI 優先。仮想拡張ポイントを先に作らない。実際の拡張要求が来たときに整える。

#### L — Liskov Substitution Principle

派生型は基底型と置換可能でなければならない。

- サブクラスが基底の事前条件を強めない。事後条件を弱めない。
- インターフェースの「契約」を継承で破壊しない。

#### I — Interface Segregation Principle

クライアントは使わないメソッドへの依存を強制されてはならない。

- 巨大なインターフェースを作らない。
- 利用者ごとに必要なメソッドだけのインターフェースを切る。

#### D — Dependency Inversion Principle

上位モジュールは下位モジュールに依存しない。両者は抽象に依存する。

- ビジネスロジックは具体的なフレームワーク・DB・外部サービスに直接依存しない。
- インフラは抽象を実装する形で差し込む（DI）。

### アーキテクチャ指針

アーキテクチャはプロジェクトの複雑さに合わせる。正当化されない場所に層構造を強制しない。

- **スクリプト・単目的ツール・小規模プロジェクト**: フラット構成で良い。
- **UI / ロジック / IO が混ざる中規模以上のアプリ**: ビジネスロジックを IO 境界から分離する。内側のロジックから外側のフレームワークを import しない。
- **複数コンテキストを持つ複雑なドメイン**: レイヤー主軸の構成（`domain` / `application` / `interface` / `infrastructure` / `composition`）を検討し、中で bounded context に分割する。

規模によらず適用する原則:

- ビジネスロジックはフレームワークや DB の詳細に依存しない（DIP）。
- モジュール境界では抽象に依存する。具体的なインフラに直接依存しない。
- 依存方向は **外向き**: `domain` は何にも依存しない。`infrastructure` は `domain` に依存する。逆向きの依存は禁止。
- 1 つのユニットに変換・判断・永続化・表示を混ぜない（SRP）。
- 「念のため」の層を追加しない（YAGNI）。

### 環境隔離 — プロジェクト単位で依存を閉じ込める

プロジェクトが import するライブラリは、プロジェクト単位で隔離した環境で管理する。グローバルにライブラリをインストールしない。

なぜ:

- グローバル依存はプロジェクト間に**暗黙的な結合**を生む。「動いているのはたまたま X が global に入っているから」という状態は再現不能であり、Code is Truth に反する。
- 削除済みプロジェクトの依存がグローバルに残り続け、PC 全体が累積的に汚染される。
- 「動かないはずだが動いている」状態は Fail-fast の対極。

ルール:

- **ライブラリ (import するもの)** は project-local に置く。グローバルインストール禁止。
- **CLI ツール (実行ファイルとして叩くもの)** はグローバル可。ただし pipx / `uv tool install` / `cargo install` のような **isolated tool installer** 経由で入れる (tool 同士の依存を相互汚染させないため)。

言語別の具体策:

| 言語 | ライブラリ依存 | CLI ツール |
|---|---|---|
| Python | `uv venv` → `.venv`、`requirements.txt` or `pyproject.toml` で宣言 | `uv tool install <tool>` (pipx 同等) |
| Node | `package.json` + ローカル `node_modules` | `npm i -g <cli>` は CLI 用途のみ |
| Rust | `Cargo.toml` | `cargo install <bin>` |
| Go | `go.mod` | `go install <pkg>@latest` |
| Ruby | `Gemfile` + bundler | `gem install <cli>` (CLI のみ) |

隔離環境ディレクトリ (`.venv/`, `node_modules/`, `target/`, `vendor/` 等) は **必ず `.gitignore` で除外** する。

完了条件:

- 新規プロジェクトを `git clone` した別環境で、宣言された手順 (`uv venv && uv pip install -r requirements.txt` など) のみで再現できる。
- グローバル環境にプロジェクト固有のライブラリが増えない。

### コード品質

- 1 関数 = 1 つの仕事。巧妙な名前より明瞭な短い名前。
- 浅いネスト、早期 return。
- 型注釈をサポートする言語では、公開関数に型注釈を付ける。
- 不明瞭なコードをコメントで補わず、コード自体を書き直す。
- デッドコード・コメントアウトされたブロック・担当者不明の TODO を残さない。

## Part 2: 開発フロー — TDD 中心

`Red → Green → Refactor` を軸とする。
このループを、調査・前提検証・小さなステップ計画で挟む形で運用する。

### 1. タスクを理解する

コードに触れる前に:

- 依頼内容を 1 文で言い換える。
- スコープ**外**を明示する。
- 成功条件を明示する。これは後でテストの形に翻訳される。
- 不明点があれば質問する、または仮定を明示する。

これができないなら、止まって確認する。

### 2. 既存コンテキストを読む

新しいコードを書く前に、次を読む:

- これから編集するファイル。
- そのファイルを呼び出している / 呼び出されているファイル。
- 関連するテスト（既存の振る舞い仕様）。
- 作業対象エリアのディレクトリ構成。

読んでいないコードを推測で扱わない。
AI エージェントには、まず **read-only モードで探索**させ、テスト計画を提示させてから実装に移らせる（Anthropic 内部の運用に倣う）。

### 3. 最小ステップを計画する

タスクを、次の条件を満たす変更単位に分解する:

- 責務が 1 つ。
- 入力 → 出力の契約が明確。
- 失敗時の挙動が明確。
- **テストとして書ける**。

これらの形で記述できないなら、設計がまだ不明確。実装前に整える。
自明でないタスクでは、実装前にユーザーに計画を共有する。

### 4. 前提を検証する

外部に依存する行を書く前に、存在を確認する:

- **外部ライブラリ呼び出し** → 推測ではなく実際のシグネチャを確認する。
- **他ファイルからの import** → そのファイルを開き、シンボルが存在するか確認する。
- **型の形** → 型定義を読む。フィールドを捏造しない。
- **設定 / 環境変数** → 設定済みか確認するか、意図的に追加する。

幻覚（ハルシネーション）による呼び出しは AI バグの最大の発生源。
このステップは数秒で済むが、数時間のデバッグを防ぐ。

### 5. テストを書く — Red

実装より先にテストを書く。これは規律であり、AI に任せると逆をやる。

書くべきテストの順序:

1. **正常系の最小ケース** — 一番単純な入力 → 期待出力。
2. **境界条件** — 空・null・最小・最大・上限。
3. **失敗パス** — 不正入力でどう fail するか。
4. **副作用** — IO / 状態遷移があるなら、その契約。

テストを書いたら、**まず実行して fail することを確認**する。
fail しないテストは、テスト自身がバグっているか、想定の振る舞いがすでに存在している。

failing テストはコミットする。これが「ここから何を作るか」の executable contract になる。

### 6. 実装する — Green

そのテストを通すための **最小** のコードだけ書く。

- 1 変更 = 1 責務。
- タスクの主旨でない限り、既存のコードスタイル・構造に合わせる。
- テストで要求されていない機能を実装しない（YAGNI）。
- 「動かすため」の try/except やフォールバックを追加しない（Fail-fast）。
- 公開契約は維持する（タスクが契約変更でない限り）。

テストが通った瞬間、そのステップは完了。次に進む。

### 7. 整える — Refactor

テストが通っている状態で、構造を改善する。

- 重複の抽出（Rule of Three を満たすなら）。
- 責務の分離（SRP に違反しているなら）。
- 名前の改善。
- デッドコード・未使用パラメータの削除。

**Refactor 中はテストを変更しない**。テストが安全網。
テストが通らなくなったら、そのリファクタは振る舞いを変えている。戻す。

### 8. バグは根本を修正する

バグが発生したら、TDD ループに乗せる:

1. **Red**: バグを再現する failing テストを書く。これが回帰テストになる。
2. **Green**: 原因のある層 / 責務を特定し、根本を修正する。テストが通る。
3. **Refactor**: 同類のバグが入りにくい構造に整える。

症状対応の兆候: 壊れた状態を隠すための sort / filter / retry / fallback の追加、エラーを握り潰す catch、共通コードを触らないための処理の複製。
これらをしている自分に気づいたら、修正すべき場所はおそらく別にある。

### 9. 変更をクローズする

完了を報告する前に:

- スコープが依頼と一致している。おまけのリファクタや機能がない。
- 投機的コード・未使用パラメータ・不要な分岐がない。
- テストが存在し、パスしている。
- 環境が動かない場合は、その旨を明示する。動作したと主張しない。
- アーキテクチャと依存方向が保たれている。
- バグ修正の場合、Red フェーズで書いた回帰テストが残っている。

## Part 3: AI 協働ルール

### AI の既定の癖を理解する

AI コーディングエージェント（Claude Code / Cursor など）は強力だが、放っておくと次のことをやる:

- **実装を先に書く**。テストは後付けで書きたがる → 明示的に「テストを先に書け」と指示する。
- **ハルシネーションで API を呼ぶ**。存在しない関数 / フィールドを使う → 前提検証を強制する。
- **スコープを広げる**。依頼にない「ついでの改善」を加える → 1 責務に押し戻す。
- **症状を隠す**。テストを通すために fallback / try-except を入れる → 根本修正を要求する。
- **既存パターンを無視する**。新しいパターンを持ち込みたがる → コードベース既存のパターンを優先させる。

これらの癖を前提に、**規律を仕組みで強制する**のがガイドラインの役割。

### Subagent でフェーズを分離する

単一のコンテキストで TDD を回すと、test writer の意図が implementer に汚染される（「結局この実装を通すためにこのテストを書きたい」というバイアス）。

複雑なタスクでは、**フェーズごとに別 subagent / 別セッション**を使う:

- **Subagent A — Test writer**: 仕様だけを見てテストを書く。実装を見ない。
- **Subagent B — Implementer**: failing テストを与えられ、それを通す実装だけ書く。なぜそのテストが書かれたかは知らない。
- **Subagent C — Refactorer**: 通ったテストを安全網に、構造改善する。テストは変更しない。

これでコンテキスト汚染を防ぎ、TDD が形式的でなく実質的に機能する。

### Plan → Execute の二段構え

Cursor Composer 2 / Claude Code に倣い、エージェントには次の流れで動かせる:

1. **Explore (read-only)** — コードベースを read-only で探索し、関連ファイル・既存パターンを把握する。
2. **Plan** — 変更計画とテスト計画を提示する。ここで人間がレビューする。
3. **Execute** — 承認された計画に沿って Red-Green-Refactor を回す。

Plan を飛ばして Execute に入らせない。Plan が共有されない実装は、後で剥がすコストが大きい。

### AI に作業を依頼するときの指示テンプレート

タスクを AI に投げるときは、必ず以下を明示する:

```
目的:        なぜこの変更が必要か
責務:        この変更が引き受けるのは何か（1 つに絞る）
入出力:      何を受け取り、何を返すか / 何を変更するか
許可する依存: 触ってよいモジュール / 触ってはいけないモジュール
テスト観点:  どんなテストで完了を確認するか
完了条件:    何が満たされたら done か
```

これが書けないなら、依頼者の側で設計が固まっていない。AI に投げる前に整える。

### AI がやってよいこと

- 小さな実装の提案。
- 設計の改善提案。
- テストおよびテスト観点のドラフト。
- 既存構造の要約・調査・前提検証。
- TDD の Red-Green-Refactor ループの実行。

### AI がやってはいけないこと

- 存在確認していない API・クラス・ファイル・挙動を捏造する。
- タスクと無関係なリファクタを行う。
- 投機的なオプション・分岐・将来対応を追加する（YAGNI 違反）。
- フォールバックや広すぎる例外 catch で問題を隠す（Fail-fast 違反）。
- テストを通すためにテスト自体を緩める。
- 未検証のコードを完了扱いする。
- 指示なしにアーキテクチャ方針を変更する。
- 別形式のドキュメント仕様を増やす（Code is Truth 違反）。

## Part 4: テスト方針

テストはカバレッジ数値を稼ぐためではなく、**振る舞いを executable contract として固定する** ために書く。

### 何をテストするか

優先度順:

1. **AI が間違えやすい箇所** — バリデーション、変換、分岐、状態遷移、境界条件、IO アダプタ。
2. **ロジック** — ユニットテスト。純粋関数 / ドメインロジックは速く厚く。
3. **IO 境界** — 統合テスト。DB / 外部 API / ファイルとの接点。
4. **ユーザーフロー** — e2e。最重要シナリオに絞る。

### テストの書き方

- **挙動をテストする。実装をテストしない**。
- 内部実装に密着した壊れやすいテストは、テストがない状態より悪い（リファクタを阻害する）。
- 1 テスト = 1 振る舞い。複数の検証を 1 ケースに詰めない。
- テスト名は「何をしたら何になる」を述べる。`test_user_login` ではなく `login_with_invalid_password_returns_401`。
- バグ修正は、Red フェーズで書いた回帰テストが残っていない限り未完了。

---

## 迷ったとき

1. タスクを 1 文で述べられるか? ないなら、明確化する。
2. 責務は 1 つか? ないなら、分割する。
3. **failing テストを書けるか?** ないなら、要求がまだ曖昧。
4. 結果を検証できるか? できないなら、検証手段を先に定義する。
5. これは根本原因の修正か? ないなら、もう一段深掘る。
6. 既存パターンに合わせているか? 外れているなら、理由があるはず。
7. SOLID / KISS / YAGNI に反していないか? 反しているなら、その正当化を言語化できるか。
