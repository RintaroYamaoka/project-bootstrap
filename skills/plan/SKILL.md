# /plan — 探索 → 計画 → 提示

このスキルは、自明でない作業を始める前に **構造化された計画書** を出力するためのもの。

**鉄則**: このスキルの実行中は **実装を行わない**。`Edit` / `Write` ツールを使わない。`Read` / `Grep` / `Glob` だけで探索し、計画を出力したら停止する。ユーザーの承認を得てから初めて実装フェーズに移行する。

## 起動条件

以下の作業に入る前に呼び出す:

- **新機能の追加**
- **バグ修正** (再現条件が曖昧、または修正範囲が不明確な場合)
- **リファクタリング**
- **調査** (コードベースの動作 / 構造を理解する必要がある)
- **自明でないタスク全般**

呼び出し方:
- ユーザーが `/plan <タスク>` と打ったとき
- 自明でない作業を依頼されたが、計画なしで実装を始めようとしているとき (自発的に呼ぶ)

呼び出さない:
- typo 修正、1 行変更、コメント追加、フォーマットなど自明な作業

## ワークフロー

### Step 1: タスクを 1 文で言い換える

依頼内容を自分の言葉で再構成する。曖昧な点があれば明確化を求める、または仮定を明示する。

### Step 2: スコープを定める

- **スコープ内**: 何をする
- **スコープ外**: 何をしないか (明示することが重要 — AI は放っておくとスコープを広げる)
- **成功条件**: 何が満たされたら done か。後でテストの形に翻訳できる粒度で書く

### Step 3: 既存コンテキストを read-only で探索する

実装に触れる前に、以下を `Read` / `Grep` / `Glob` で読む:

- 編集対象になりそうなファイル
- そのファイルを呼び出している / 呼び出されているファイル
- 関連するテスト (既存の振る舞い仕様)
- 作業対象エリアのディレクトリ構成

**この時点で `Edit` / `Write` を行わない**。読むだけ。

### Step 4: 最小ステップに分解する

タスクを、次の条件を満たす変更単位に分解する:

- 責務が 1 つ
- 入力 → 出力の契約が明確
- 失敗時の挙動が明確
- **テストとして書ける** (Red フェーズで failing テストを書ける粒度)

これらの形で記述できないステップは、設計がまだ不明確。明確化してから次に進む。

### Step 5: 前提を検証する

外部依存を確認する:

- **外部ライブラリ呼び出し** → 推測ではなく実際のシグネチャを確認 (`Read` でドキュメントや型定義)
- **他ファイルからの import** → そのシンボルが存在するか確認
- **型の形** → 型定義を読む。フィールドを捏造しない
- **設定 / 環境変数** → 設定済みか確認

検証できなかった前提は計画書の **「未検証 / リスク」** 欄に記録する。実装フェーズで検証してから使う。

### Step 6: 計画書を出力する

以下のフォーマットで出力する:

```markdown
## タスク

<1 文で言い換え>

## スコープ

**スコープ内:**
- ...

**スコープ外:**
- ...

## 成功条件

- ...

## 探索したコンテキスト

- `path/to/file.ts:LL` — <要点>
- `path/to/test.ts:LL` — <既存の振る舞い>
- ...

## ステップ

### Step 1: <名前>
- **責務**: ...
- **入出力**: ...
- **失敗時の挙動**: ...
- **テスト観点 (Red で書く failing テストの内容)**: ...

### Step 2: <名前>
...

## 検証済みの前提

- `lib.foo()` のシグネチャ: <確認済み>
- `Module.bar` の型: <確認済み>
- ...

## 未検証 / リスク

- ...

## 質問 / 仮定 (あれば)

- ...
```

### Step 7: 停止する

計画書を出力したら **実装に進まずに停止する**。
ユーザーが計画を承認するまで、コードの編集は行わない。

承認後、TDD ワークフロー (Red → Green → Refactor) に移る。各ステップは Red フェーズで failing テストを書くところから始める (詳細は `project-bootstrap` skill 参照)。

## やってはいけないこと

- このスキル中に `Edit` / `Write` を使うこと
- 計画書を出さずに実装を始めること
- ステップを「1 つの大きなブロック」として記述すること (1 ステップ = 1 責務 = 1 テスト)
- 既存コードを読まずに計画を立てること (推測で書いてはならない)
- 未検証の前提を「検証済み」として書くこと
- 計画を出した後、承認を待たずに実装を始めること

## やってよいこと

- `Read` / `Grep` / `Glob` でコードベースを探索すること
- 不明点があれば計画書の「質問」欄に記載してユーザーに質問すること
- 計画が大きすぎる場合に「これは 1 タスクには大きすぎる、分割を提案する」と伝えること
- 探索の結果、依頼内容そのものを見直すべき (例: そもそも修正不要、別箇所が原因、要件矛盾) と判断したら、それを計画書ではなく「再考の提案」として伝えること
